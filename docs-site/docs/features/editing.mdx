# Inline cell editing

Mark a column `editable` and the grid lets users edit its cells in place â€” **double-click** a cell, or focus
it and press **Enter** / **F2**. The editor is type-aware (text / number / date, and a dropdown for
`type: 'select'` or `type: 'boolean'`). **Enter** or blur commits; **Escape** cancels.

<LiveDemo>{() => <EditingDemo />}</LiveDemo>

```tsx
const columns = [
  { id: 'name', header: 'Name', accessorKey: 'name', editable: true },
  { id: 'role', header: 'Role', accessorKey: 'role', editable: true,
    type: 'select', options: [{ label: 'Admin', value: 'Admin' }, { label: 'Editor', value: 'Editor' }] },
];

<DataTable columns={columns} data={rows} editable /* per-column */ />
```

`editable` can also be a function â€” `editable: (row) => boolean` â€” to allow editing per row.

## Getting the updated data

On commit the grid calls **`processRowUpdate(newRow, oldRow)`**, then applies the row you **return** to its
data. This is the one hook you need â€” it's where you persist and where you can transform or validate:

```tsx
<DataTable
  columns={columns}
  data={rows}
  processRowUpdate={(newRow, oldRow) => {
    // newRow = oldRow with the edited field changed; return the row to apply
    return newRow;
  }}
  onProcessRowUpdateError={(err) => toast.error(String(err))}
/>
```

- **Return a row** â†’ the grid applies it (you can normalise/derive fields here, e.g. recompute a `total`).
- **Throw / reject** â†’ `onProcessRowUpdateError` fires and the edit is **reverted**.
- **No `processRowUpdate`** â†’ edits update the grid's **local data** directly (handy for prototypes).

Prefer to listen rather than intercept? Use `onDataStateChange` for view state, or read the data at any time
from [`apiRef`](#read--write-data-imperatively).

## Which field gets written

The grid writes to the column's **`accessorKey`** (falling back to `id`), so a column whose `id` differs from
its `accessorKey` still updates the right field. **Dot-nested** accessor keys are deep-set immutably â€” only the
path is cloned, siblings are preserved:

```tsx
// accessorKey: 'address.city' â†’ newRow.address.city is updated (address.zip untouched)
{ id: 'city', header: 'City', accessorKey: 'address.city', editable: true }
```

The edited row keeps its **identity** via your `getRowId` (or the `idKey` prop, default `'id'`) â€” that's the id
passed to `apiRef.current.data.updateRow`, so make it stable if you persist.

## Server-side persistence

`processRowUpdate` may be **async** â€” `await` your API and return the saved row (so server-computed fields flow
back into the grid). Throw to reject and revert:

```tsx
processRowUpdate={async (newRow, oldRow) => {
  const saved = await api.users.update(newRow.id, newRow); // PATCH/PUT
  return saved;                                             // grid applies the server's row
}}
```

This works in [server data mode](/features/server-side) too: the returned row updates the **loaded page** in
place. If a server-side edit changes sorting/filtering/paging membership, trigger a refetch with
`apiRef.current.data.refresh()` after it resolves.

## Validation

Validate inside `processRowUpdate` and **throw** to reject â€” the grid reverts and surfaces the error:

```tsx
processRowUpdate={(newRow) => {
  if (!newRow.email.includes('@')) throw new Error('Invalid email');
  return newRow;
}}
onProcessRowUpdateError={(err) => setError(String(err))}
```

## Read & write data imperatively

Drive data from your own UI through [`apiRef`](/api/api-ref) â€” the same `data` API the editor commits through:

```tsx
apiRef.current?.data.updateRow(rowId, { role: 'Admin' });          // patch one row
apiRef.current?.data.updateField(rowId, 'status', 'active');       // patch one field
apiRef.current?.data.getRowData(rowId);                            // read a row
apiRef.current?.data.getAllData();                                 // read all rows
apiRef.current?.data.updateMultipleRows([{ rowId, data: {â€¦} }]);   // batch
```

## Type-aware editors

| Column `type` | Editor |
| --- | --- |
| `text` (default) | text input |
| `number` | numeric input (commits a `Number`) |
| `date` | date input (keeps `Date` vs string to match the source) |
| `select` (with `options`) | dropdown â€” commits on choice |
| `boolean` | True / False dropdown |

## Props

| Prop | Type | Description |
| --- | --- | --- |
| `editable` (column) | `boolean \| (row) => boolean` | Make a column's cells editable |
| `processRowUpdate` | `(newRow, oldRow) => Row \| Promise<Row>` | Commit hook â€” return the row to apply (or throw to revert) |
| `onProcessRowUpdateError` | `(error) => void` | Fires when `processRowUpdate` throws/rejects |

:::note
Editing is single-cell, and in [tree data](/features/tree-data) only top-level rows are editable. **Whole-row
edit mode** (open a row's cells together with explicit Save / Cancel) and **custom cell editors** are the next
editing enhancements.
:::
